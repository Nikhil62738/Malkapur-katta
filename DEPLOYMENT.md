# Deployment Guide — Netlify (frontend) + Render (backend)

Malkapur Katta is two apps in one repo:

- **Frontend** — Vite + React SPA at the repo root → deploy to **Netlify**.
- **Backend** — Express + MongoDB API in `server/` → deploy to **Render**.
- **Database** — use **MongoDB Atlas** (free tier) since Render has no built-in MongoDB.

Config files included: `netlify.toml` (frontend) and `render.yaml` (backend blueprint).

---

## 1. MongoDB Atlas (database)

1. Create a free cluster at https://www.mongodb.com/atlas .
2. Create a database user (username + password).
3. Under **Network Access**, allow `0.0.0.0/0` (so Render can connect).
4. Copy the connection string, e.g.
   `mongodb+srv://USER:PASSWORD@cluster0.xxxx.mongodb.net/malkapur_katta?retryWrites=true&w=majority`
   (Keep the `/malkapur_katta` database name before the `?`.)

---

## 2. Backend on Render

1. Push this repo to GitHub.
2. In Render: **New +** → **Blueprint**, pick the repo. Render reads `render.yaml`.
3. Fill in the secret env vars when prompted:
   - `MONGODB_URI` — your Atlas string from step 1.
   - `CORS_ORIGIN` — your Netlify URL (you'll get it in step 3); you can set a placeholder now and update it after the frontend deploys. Example: `https://malkapur-katta.netlify.app`
   - `ADMIN_EMAIL` — e.g. `admin@malkapurkatta.com`
   - `ADMIN_PASSWORD` — a strong password.
   - `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET` — from your Cloudinary dashboard (needed for image/video uploads).
   - `JWT_SECRET` is auto-generated by Render.
4. Deploy. Your API will be at something like `https://malkapur-katta-api.onrender.com`.
5. **Seed the database once**: open the service's **Shell** tab in Render and run:
   ```bash
   npm run seed
   ```
   This creates the super-admin account and loads the default content. (The login also auto-promotes the configured `ADMIN_EMAIL` to super admin, so you can log in even before seeding.)
6. Verify: visit `https://<your-api>.onrender.com/api/health` — it should return `{"ok":true}`.

> **Render free tier note:** free web services sleep after ~15 minutes of inactivity and take ~30–50s to wake on the next request. That cold start is the main cause of "too much loading". To avoid it, either upgrade to a paid instance, or ping `/api/health` every ~10 minutes with a free uptime monitor (e.g. UptimeRobot / cron-job.org).

---

## 3. Frontend on Netlify

1. In Netlify: **Add new site** → **Import an existing project** → pick the repo.
2. Netlify reads `netlify.toml` (build = `npm install && npm run build`, publish = `dist`). If asked, leave the **base directory** empty (repo root).
3. Add the environment variable **Site settings → Environment variables**:
   - `VITE_API_BASE_URL` = your Render API URL, e.g. `https://malkapur-katta-api.onrender.com`
     (No trailing slash. The app appends `/api` itself.)
4. Deploy. You'll get a URL like `https://malkapur-katta.netlify.app`.
5. Go back to Render and set `CORS_ORIGIN` to this exact Netlify URL, then redeploy the backend (or trigger "Clear cache & deploy").

---

## 4. Verify it works

- Public site loads at the Netlify URL.
- Admin panel: go to `https://<netlify-url>/admin`, log in with `ADMIN_EMAIL` / `ADMIN_PASSWORD`.
- You should see all sidebar sections (you're the super admin) and be able to manage sub-admins at `/admin/subadmins`.
- Edits to reels, news, events, etc. persist to MongoDB.

---

## 5. Custom domain (optional)

- Point your domain to Netlify (frontend). 
- Update `CORS_ORIGIN` on Render to include the custom domain (comma-separated if you keep both), and update `VITE_API_BASE_URL` only if you also move the API to a custom domain.

## Fixing `npm error Missing script: "start"` on Render

This error means Render ran `npm start` at the repository **root**, but the backend lives in `server/`. The root project is the frontend and has no `start` script. Fix it with **any one** of these:

### Option A — Recommended: set Root Directory to `server`

If you created the Render service manually (not via Blueprint):

1. Render dashboard → your service → **Settings**.
2. **Root Directory** = `server`
3. **Build Command** = `npm install`
4. **Start Command** = `npm start`
5. **Manual Deploy** → **Clear build cache & deploy**.

This runs `server/package.json`, which has the `start` script (`node src/index.js`).

### Option B — Keep Root Directory blank, use the root scripts

The root `package.json` now also defines a `start` script (`node server/src/index.js`). With Root Directory left empty:

1. **Build Command** = `npm install --prefix server`
2. **Start Command** = `npm start`
3. Clear build cache & deploy.

### Option C — Easiest: use the Blueprint

Delete the failing service, then **New + → Blueprint** and point it at this repo. The included `render.yaml` already sets Root Directory = `server`, the build/start commands, and the `/api/health` check automatically — you only fill in the secret env vars.

> Whichever option you pick, remember to set the env vars (`MONGODB_URI`, `CORS_ORIGIN`, `ADMIN_EMAIL`, `ADMIN_PASSWORD`, Cloudinary keys). After the first deploy, open the Render **Shell** and run `npm run seed` once (use `npm run seed` from `server/` for Option A, or `npm --prefix server run seed` for Option B).
